## 1. Цели и предпосылки

### 1.1 Зачем ML? Сравнение с текущим ручным процессом

**Бизнес-цель:** Оптимизировать управление товарными запасами и логистикой путём точного прогнозирования спроса на товары в каждой точке продаж сети супермаркетов.

**Текущий процесс (AS-IS):** Заказы товаров формируются вручную менеджерами магазинов на основе интуиции и визуального осмотра полок. Менеджер оценивает остатки, вспоминает «обычные» объёмы и формирует заявку в Excel или 1С. Подробная диаграмма текущего процесса: [BPMN AS-IS](diagrams/bpmn-as-is.md).

**Проблемы ручного процесса:**

| Проблема | Влияние | Масштаб |
|----------|---------|---------|
| Избыток скоропортящихся товаров | Списания (молоко, хлеб, овощи) | 2–4% выручки |
| Дефицит популярных товаров | Упущенные продажи (out-of-stock) | 3–5% выручки |
| Неучёт факторов спроса | Менеджер не может учесть 50+ факторов | Все 10 000 SKU |
| Невоспроизводимость | Качество зависит от опыта конкретного менеджера | 500 магазинов |
| Временные затраты | 1–2 часа в день на формирование заявки | 500–1000 чел·ч/день |

**Почему ML, а не правила/эвристики:**

- **Масштаб данных:** 5 млн комбинаций (10 000 товаров × 500 магазинов) — невозможно написать правила для каждой.
- **Нелинейные зависимости:** Влияние погоды, промоакций, праздников, каннибализации товаров — факторы взаимодействуют нелинейно.
- **Адаптивность:** ML-модель автоматически адаптируется к изменениям (новые товары, смена трендов, открытие/закрытие магазинов).
- **Рентабельность:** Даже 1% улучшение точности прогноза при обороте сети в десятки миллиардов рублей окупает инвестиции в ML многократно.

Целевой процесс после внедрения: [BPMN TO-BE](diagrams/bpmn-to-be.md).

### 1.2 Бизнес-требования и ограничения

**Бизнес-требования:**

| # | Требование | Приоритет |
|---|-----------|-----------|
| BR-1 | Снижение списаний скоропортящихся товаров на ≥20% | Критический |
| BR-2 | Снижение out-of-stock на ≥15% | Критический |
| BR-3 | Автоматическое формирование заявок с возможностью корректировки менеджером | Критический |
| BR-4 | Сокращение времени менеджера на формирование заказа с 1–2 часов до 10–15 минут | Высокий |
| BR-5 | Учёт промоакций в прогнозе | Высокий |
| BR-6 | Визуализация прогнозов и метрик для менеджеров | Средний |
| BR-7 | Возможность масштабирования на всю сеть (500 магазинов) | Высокий |

**Ограничения:**

- **Команда:** 3 человека — необходимо использовать managed-сервисы и open-source решения, минимизируя кастомную инфраструктуру.
- **Данные:** Доступна история продаж за 1 год. Для некоторых товаров (новинки) истории может быть недостаточно — необходимы fallback-стратегии (cold start problem).
- **Интеграция:** Существующая ERP-система (1С/SAP) не может быть заменена — только интеграция через API/CDC.
- **Человек в контуре:** Менеджер остаётся финальным звеном принятия решений — система рекомендует, не диктует.
- **Бюджет инфраструктуры:** Ориентир на self-hosted решения (MinIO, ClickHouse) с возможностью миграции в облако.

### 1.4 Процесс пилота и критерии успеха

Пилот проводится перед полномасштабным внедрением для валидации гипотезы и снижения рисков.

**Этапы пилота:**

```
1. Подготовка данных       (2 недели)  — загрузка истории, EDA, качество данных
2. Разработка MVP модели   (3 недели)  — baseline + первая итерация
3. A/B-тест на пилоте      (4–6 недель) — 10–20 магазинов
4. Анализ результатов      (1 неделя)  — метрики, выводы, решение о масштабировании
```

**Критерии успеха пилота (go/no-go):**

| Метрика | Порог | Обоснование |
|---------|-------|-------------|
| WMAPE по скоропортящимся | ≤25% | Критическая категория для списаний |
| WMAPE по всем товарам | ≤30% | Общая точность прогноза |
| Снижение списаний | ≥10% vs контрольная группа | Бизнес-эффект |
| Снижение OOS | ≥8% vs контрольная группа | Бизнес-эффект |
| Доля заявок без корректировок | ≥60% | Доверие менеджеров к системе |

### 1.5 Scope MVP vs техдолг

**В scope MVP:**

- Прогнозирование спроса (LightGBM, ежедневный batch)
- Автогенерация заявок (простая логика: прогноз − остаток + safety stock)
- Веб-интерфейс для менеджеров (просмотр прогнозов, корректировка заявок)
- Базовый мониторинг (Grafana: метрики модели, системные метрики)
- Интеграция с ERP для получения данных о продажах и остатках
- Пилот на 10–20 магазинах

**Техдолг (после MVP):**

- Online inference (сейчас — batch)
- Модели для cold start (новые товары без истории)
- A/B-тестирование моделей в production
- Автоматическое переобучение по дрифту
- Каннибализация и эластичность спроса
- Оптимизация логистики (маршруты, комплектация)
- Мобильное приложение для менеджеров
- Прогноз на уровне часа (для fresh-категорий)
- Feature Store с полноценным online serving (Redis)

## 2. Методология

### 2.1 Тип задачи

**Time series forecasting** на уровне гранулярности **товар × магазин × день**.

| Параметр | Значение |
|----------|----------|
| Целевая переменная | `qty_sold` — количество проданных единиц товара |
| Гранулярность | 1 день |
| Горизонт прогноза | 1–7 дней (MVP), до 14 дней (v2) |
| Количество временных рядов | ~5 000 000 (10 000 товаров × 500 магазинов) |
| Длина ряда | ~365 точек (1 год истории) |

**Особенности задачи:**

- **Иерархическая структура:** Прогнозы должны быть согласованы по иерархии (SKU → подкатегория → категория → магазин → регион → сеть).
- **Intermittent demand:** Часть товаров продаётся редко (< 1 шт/день) — стандартные метрики неприменимы, нужны специализированные подходы (Croston's method).
- **Промо-эффект:** Промоакции создают выбросы в данных, которые нужно моделировать отдельно.
- **Каннибализация:** Скидка на один товар может снижать продажи аналогов.
- **Новые товары:** Cold start — нет истории для обучения, нужен fallback на аналоги.

**Подход к моделированию:**

Основной алгоритм — **LightGBM** (gradient boosting) с фичами временных рядов. Выбор обоснован:
- Лучшая производительность на табличных данных с гетерогенными фичами
- Нативная работа с категориальными переменными
- Быстрое обучение и инференс (критично при 5 млн рядов)
- Интерпретируемость через feature importance

Дополнительный алгоритм — **Prophet** для товаров с выраженной сезонностью и трендами, где декомпозиция ряда даёт лучшие результаты.

### 2.2 Необходимые данные

Модель данных: [ER-диаграмма](diagrams/data-model.md).

| Источник | Данные | Объём | Обновление |
|----------|--------|-------|------------|
| POS-системы | История продаж (транзакции) | 2.67 млрд строк/год (~7.3 млн/день) | Real-time через Kafka |
| ERP (1С/SAP) | Остатки, номенклатура, цены, поставщики | ~5 млн записей остатков/день | CDC (Debezium) |
| ERP | Промоакции (товар, период, скидка) | ~500–1000 акций/месяц | CDC |
| Внешний API | Погода (температура, осадки, ветер) | По городам присутствия | Ежедневно |
| Справочник | Календарь (праздники, выходные, школьные каникулы) | Статический, обновляется 1 раз/год | Ежемесячно |

**Фичи для модели:**

| Группа | Фичи | Пример |
|--------|-------|--------|
| Лаговые | Продажи за предыдущие N дней | `sales_lag_1`, `sales_lag_7`, `sales_lag_14` |
| Скользящие средние | Среднее / медиана за окно | `sales_ma_7`, `sales_ma_30`, `sales_ma_90` |
| Календарные | День недели, месяц, праздник | `day_of_week`, `is_holiday`, `is_payday_week` |
| Ценовые | Текущая цена, скидка, ценовой индекс | `price`, `discount_pct`, `price_vs_avg` |
| Промо | Наличие акции, тип акции, день акции | `promo_flag`, `promo_day_num`, `promo_type` |
| Остатки | Текущий остаток, дней запаса | `qty_on_hand`, `days_of_supply` |
| Погодные | Температура, осадки | `temp_max`, `precipitation` |
| Товарные | Категория, срок годности, вес | `category`, `shelf_life_days`, `is_perishable` |
| Магазинные | Формат, площадь, регион | `store_format`, `area_sqm`, `city` |

### 2.3 ML-метрики и связь с бизнес-метриками

**ML-метрики:**

| Метрика | Формула | Назначение |
|---------|---------|------------|
| **MAPE** | `mean(\|actual − predicted\| / actual) × 100%` | Интуитивная метрика ошибки в %. Не подходит для товаров с нулевыми продажами |
| **WMAPE** | `sum(\|actual − predicted\|) / sum(actual) × 100%` | Взвешенная MAPE — основная метрика. Устойчива к нулевым продажам, учитывает вклад каждого товара пропорционально объёму |
| **MAE** | `mean(\|actual − predicted\|)` | Средняя абсолютная ошибка в единицах товара. Интерпретируема для менеджеров: «модель ошибается в среднем на X штук» |

**Связь ML-метрик с бизнес-метриками:**

```
ML-метрика                    Бизнес-метрика
──────────────────────────────────────────────────────
WMAPE ↓ (точность прогноза)  → % списаний ↓
                              → % out-of-stock ↓
                              → упущенная выручка ↓

Bias (смещение прогноза)      → Если модель завышает → списания ↑
                              → Если модель занижает → OOS ↑

Coverage (% товаров           → Доля автоматизированных
с приемлемым WMAPE)             заявок ↑
```

**Целевые значения:**

| Метрика | Baseline (ручной) | Цель MVP | Цель v2 |
|---------|-------------------|----------|---------|
| WMAPE (скоропортящиеся) | ~40–50% | ≤25% | ≤20% |
| WMAPE (все товары) | ~35–45% | ≤30% | ≤22% |
| % списаний | 3–4% | ≤2.5% | ≤1.5% |
| % OOS | 5–8% | ≤4% | ≤2.5% |
| Доля автозаявок без корректировок | 0% | ≥60% | ≥80% |

**Как измеряется baseline (ручной процесс):**

Для корректного сравнения необходимо зафиксировать текущее качество ручных заказов. Методика: в течение 2 недель до пилота собирать фактические заказы менеджеров, сравнивать с реальными продажами и вычислять WMAPE «ручного прогноза» (заказ менеджера ≈ его прогноз спроса).

### 2.4 Риски на этапе моделирования

| # | Риск | Вероятность | Влияние | Митигация |
|---|------|-------------|---------|-----------|
| R-1 | Низкое качество данных (пропуски, дубли, аномалии в POS) | Высокая | Критическое | EDA + пайплайн очистки, мониторинг качества данных, алерты при аномалиях |
| R-2 | Cold start для новых товаров | Высокая | Среднее | Fallback на аналоги (по подкатегории/категории), экспертные правила для первых 4 недель |
| R-3 | Недостаточная длина истории (1 год) | Средняя | Среднее | Начать с категорий, где сезонность укладывается в 1 год; запросить данные за 2–3 года если доступны |
| R-4 | Каннибализация товаров при промо | Средняя | Среднее | В MVP не моделируем, добавляем в v2; на пилоте оцениваем масштаб эффекта |
| R-5 | Overfitting на малых рядах | Средняя | Среднее | Кросс-валидация (time series split), регуляризация, global model (один на все ряды) |
| R-6 | Data leakage (утечка будущего в обучение) | Средняя | Критическое | Strict time-based split, Feature Store с point-in-time correctness, code review |
| R-7 | Концептуальный дрифт (смена паттернов спроса) | Низкая | Высокое | Мониторинг дрифта, регулярное переобучение, алерты при деградации метрик |

## 3. Подготовка пилота

### 3.1 Способ оценки

**Дизайн эксперимента: A/B-тест на уровне магазинов.**

| Параметр | Значение |
|----------|----------|
| Тип эксперимента | A/B-тест (кластерная рандомизация по магазинам) |
| Тестовая группа | 10–20 магазинов (ML-прогнозы + автозаявки) |
| Контрольная группа | 10–20 магазинов (текущий ручной процесс) |
| Длительность | 4–6 недель |
| Единица рандомизации | Магазин (не товар — во избежание spillover-эффектов) |

**Выбор магазинов для пилота:**

Магазины выбираются по принципу стратифицированной рандомизации для обеспечения репрезентативности:

| Критерий стратификации | Варианты |
|------------------------|----------|
| Формат магазина | Мини, стандарт, супер |
| Регион | 2–3 разных города |
| Оборот | Высокий, средний, низкий |
| Качество данных POS | Проверенное (минимум аномалий) |

Тестовая и контрольная группы формируются так, чтобы распределение по критериям было схожим (matched pairs design).

### 3.2 Критерии успеха

**Количественные критерии (go/no-go):**

| # | Метрика | Метод измерения | Порог для GO | Порог для NO-GO |
|---|---------|-----------------|--------------|-----------------|
| 1 | WMAPE (скоропортящиеся) | Прогноз vs факт, ежедневно | ≤25% | >35% |
| 2 | WMAPE (все товары) | Прогноз vs факт, ежедневно | ≤30% | >40% |
| 3 | Δ списаний (тест vs контроль) | Сумма списаний за период | ≥−10% | <−3% |
| 4 | Δ OOS (тест vs контроль) | Доля SKU с нулевым остатком | ≥−8% | <−3% |
| 5 | Доля заявок без корректировок | Кол-во approved / total | ≥60% | <40% |

**Качественные критерии:**

- Менеджеры пилотных магазинов дают положительную обратную связь (NPS ≥ 7)
- Нет критических инцидентов (полная неработоспособность >4 часов)
- Время на формирование заявки снизилось (замер по логам сессий)

**Зона неопределённости (между GO и NO-GO):** Если результаты попадают между порогами — продление пилота на 2–4 недели с дополнительными итерациями модели.

## 4. Внедрение в production

### 4.1 Архитектура решения

Архитектура PrismFlow построена по принципу разделения ответственности: каждый компонент решает одну задачу и взаимодействует с другими через явные интерфейсы (REST API, Kafka, SQL).

[Архитектура системы](diagrams/architecture.md)

Система состоит из следующих слоёв:

1. **Data Sources** — POS-системы магазинов, ERP, внешние API (погода, календарь)
2. **Ingestion Layer** — Kafka + Debezium CDC для надёжного приёма данных
3. **Data Lake / DWH** — S3 (MinIO) + ClickHouse для хранения и аналитики
4. **ML Platform** — Feature Store (Feast) + Airflow + MLflow для ML-пайплайнов
5. **Model Serving** — FastAPI-сервис инференса + генератор заявок
6. **Application Layer** — PostgreSQL + Order Management API + уведомления
7. **Frontend** — React-дашборд для менеджеров
8. **Monitoring** — Grafana + Prometheus + ML-мониторинг

Детальные диаграммы:
- Компоненты системы: [UML Components](diagrams/uml-components.md)
- Процесс формирования прогноза: [UML Sequence](diagrams/uml-sequence.md)

### 4.2 Инфраструктура и масштабируемость

**Текущие требования к ресурсам (MVP на пилот):**

| Компонент | CPU | RAM | Disk | Кол-во |
|-----------|-----|-----|------|--------|
| Kafka (broker) | 4 vCPU | 8 GB | 500 GB SSD | 1 (3 для prod) |
| ClickHouse | 8 vCPU | 32 GB | 2 TB SSD | 1 |
| PostgreSQL | 4 vCPU | 16 GB | 200 GB SSD | 1 |
| MinIO (S3) | 2 vCPU | 4 GB | 5 TB HDD | 1 |
| Airflow | 4 vCPU | 8 GB | 50 GB | 1 |
| MLflow | 2 vCPU | 4 GB | 100 GB | 1 |

| Grafana + Prometheus | 2 vCPU | 4 GB | 50 GB | 1 |

**Стратегия масштабирования при росте до 1000 магазинов:**

- **Kafka:** Добавление брокеров и партиций (горизонтальное масштабирование).
- **ClickHouse:** Переход на кластер с шардированием по `store_id`.
- **ForecastService:** Горизонтальное масштабирование (stateless, за балансировщиком).
- **Training:** Параллельное обучение на нескольких воркерах (по регионам/категориям).

**Развёртывание:**

MVP развёртывается на bare-metal или VM-серверах с Docker Compose. При масштабировании — миграция на Kubernetes для оркестрации контейнеров. CI/CD через GitHub Actions: тесты → сборка образов → деплой на staging → ручное подтверждение → production.

### 4.3 SLA, латентность, batch vs online

**Режим работы: Batch (ежедневный).**

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| Режим инференса | Batch (ежедневно, 02:00 MSK) | Заявки формируются 1 раз в день; real-time не нужен на MVP |
| Время формирования прогноза | ≤2 часов (all stores × all products) | Прогнозы должны быть готовы к 06:00, когда менеджеры приходят на работу |
| Окно доступности API | 06:00–23:00 MSK (рабочие часы) | Менеджеры работают днём; ночью — batch-обработка |
| SLA API | 99.5% в рабочие часы | Допустим простой до 4 ч/месяц; заявки можно подтвердить на следующий день |
| Латентность API (p95) | ≤500 мс | Комфортная работа в веб-интерфейсе |
| Латентность API (p99) | ≤1000 мс | Допустимо для аналитических запросов |
| RPO (Recovery Point Objective) | 24 часа | Потеря данных за 1 день — восстановление из ERP |
| RTO (Recovery Time Objective) | 4 часа | Максимальное время восстановления |

**Переход к online-инференсу (v2):**

В будущем для fresh-категорий (хлеб, молоко) может потребоваться intra-day прогноз (обновление 2–3 раза в день) для корректировки заявок. Архитектура (Feature Store с online serving, FastAPI сервис) уже подготовлена для этого перехода.

### 4.4 Риски при внедрении в production

| # | Риск | Вероятность | Влияние | Митигация |
|---|------|-------------|---------|-----------|
| P-1 | Сопротивление менеджеров | Высокая | Критическое | Постепенное внедрение, обучение, демонстрация результатов на пилоте. Менеджер остаётся «финальным звеном» |
| P-2 | Сбой пайплайна (прогнозы не сформированы к утру) | Средняя | Высокое | Алерты, автоматический ретрай, fallback на прогноз предыдущего дня, ручной режим как backup |
| P-3 | Деградация модели без переобучения | Средняя | Высокое | Мониторинг метрик, автоалерт при WMAPE > порога, регулярное переобучение (еженедельно) |
| P-4 | Перебои в интеграции с ERP | Средняя | Высокое | CDC через Kafka (буферизация), мониторинг лага, алерт при задержке > 1 часа |
| P-5 | Рост нагрузки при масштабировании на 500 магазинов | Средняя | Среднее | Нагрузочное тестирование на этапе пилота (эмуляция full-scale), горизонтальное масштабирование |
| P-6 | Data leakage в production | Низкая | Критическое | Strict feature pipeline (Feature Store), автоматические тесты на temporal consistency |
| P-7 | Ошибка модели на массовом уровне (все магазины получают неверный прогноз) | Низкая | Критическое | Санити-проверки перед публикацией прогнозов (отклонение > 3σ → блокировка + алерт), canary deployment моделей |
| P-8 | Потеря ключевого члена команды (bus factor = 3) | Низкая | Критическое | Документация, code review, парное программирование, cross-training |
